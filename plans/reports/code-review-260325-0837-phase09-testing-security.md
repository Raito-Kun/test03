## Critical Issues

### 1. Backend Dockerfile: `COPY ../shared` breaks Docker build context

**File:** `packages/backend/Dockerfile` line 6
```dockerfile
COPY ../shared ./packages/shared
```

Docker `COPY` cannot reference paths outside the build context. Since `docker-compose.prod.yml` sets `context: ./packages/backend`, the `../shared` path is unreachable and the build will fail.

**Fix options:**
- (A) Move `context` to repo root and adjust paths:
  ```yaml
  backend:
    build:
      context: .
      dockerfile: packages/backend/Dockerfile
  ```
- (B) Use a root-level Dockerfile that copies both `packages/backend` and `packages/shared`.

**Impact:** Build is broken for production Docker deployment.

---

### 2. server.listen() fires on test import

**File:** `packages/backend/src/index.ts` lines 110-120

When `tests/helpers.ts` imports `app` from `../src/index`, the module-level `server.listen()` runs unconditionally. This binds port 4000 during tests. It works today because only one test file imports `helpers.ts` (via vitest default parallelism per file), but it is fragile and leaks resources.

**Fix:** Guard the listen call:
```typescript
if (process.env.NODE_ENV !== 'test') {
  server.listen(PORT, () => { ... });
}
```

**Impact:** High. Port collisions if test parallelism increases; resource leak during test runs (ESL daemon, reminder job also start).

## High Priority

### 3. ESL daemon and reminder job start during tests

**File:** `packages/backend/src/index.ts` lines 114-119

Even with `ESL_ENABLED=false`, `startReminderJob()` still executes during tests (it runs inside the `listen` callback which fires, per issue #2). If `listen` is guarded, this resolves itself. Verify `startReminderJob` is also safe when DB is unavailable.

### 4. JWT secret mismatch risk between test setup and jwt.ts

**Files:** `tests/setup.ts` line 9, `tests/helpers.ts` line 7, `src/lib/jwt.ts` line 11

- `setup.ts` sets `JWT_ACCESS_SECRET` as a fallback: `'test-jwt-access-secret-key-for-testing'`
- `helpers.ts` duplicates the same fallback
- `jwt.ts` uses `getRequiredEnv('JWT_ACCESS_SECRET')` which throws if the env var is missing

This works only because `setup.ts` runs first (via vitest `setupFiles`). However, the fallback is duplicated in two places. If someone changes one, tokens generated by `helpers.ts` will not match what `jwt.ts` uses, causing silent 401s in all tests.

**Fix:** Remove the fallback from `helpers.ts` and always read from `process.env.JWT_ACCESS_SECRET` (which is guaranteed set by setup.ts):
```typescript
const JWT_SECRET = process.env.JWT_ACCESS_SECRET!;
```

### 5. Frontend Docker: no `.dockerignore` coverage for source maps/tests

**File:** `packages/frontend/.dockerignore`

Only excludes `node_modules`, `dist`, `.env*`. Missing: `tests/`, `*.test.*`, `*.spec.*`, coverage folders, `.git`. Not critical since `COPY . .` followed by `npm run build` produces only `dist/`, but larger context = slower builds.

### 6. Docker Compose: frontend not on `internal` network

**File:** `docker-compose.prod.yml` lines 66-78

Frontend container is only on the `external` network, but its nginx proxies to `http://backend:4000`. The backend is on both `internal` and `external`, so DNS resolution works. However, this means the frontend-to-backend traffic flows over the `external` network. For defense-in-depth, add `internal` to the frontend networks:
```yaml
frontend:
  networks:
    - internal
    - external
```

## Medium Priority

### 7. Test assertions accept too-wide status code ranges

**Files:** `tests/endpoints.test.ts`, `tests/auth.test.ts`

Pattern like `expect([200, 500]).toContain(res.status)` means tests pass whether the endpoint succeeds or crashes. This is intentional for DB-less CI, but it means these tests cannot catch regressions -- a 500 is always acceptable.

**Recommendation:** Add a CI environment flag (e.g., `TEST_WITH_DB=true`) and tighten assertions when DB is present:
```typescript
if (process.env.TEST_WITH_DB) {
  expect(res.status).toBe(200);
} else {
  expect([200, 500]).toContain(res.status);
}
```

### 8. No test cleanup / afterAll hook

**Files:** `tests/helpers.ts`, `tests/setup.ts`

No `afterAll` closes the HTTP server, Redis connection, or Socket.IO instance. Vitest may hang or leak handles. Add:
```typescript
afterAll(async () => {
  // close server, redis, socket.io
});
```

### 9. Rate limiter `loginLimiter` not relaxed for tests

**File:** `packages/backend/src/middleware/rate-limiter.ts` line 30

`globalLimiter` and `callLimiter` have `max: isTest ? 1000 : N`, but `loginLimiter` is hardcoded to `max: 10` regardless of environment. If login tests grow beyond 10 requests, they will be rate-limited.

**Fix:** Apply the same test guard: `max: isTest ? 1000 : 10`.

### 10. Webhook security test is weak

**File:** `tests/security.test.ts` lines 82-93

The CDR webhook test only checks `status !== 404`. It does not verify any webhook authentication mechanism (e.g., shared secret, IP allowlist). If webhook auth exists, test it. If it does not exist, that is a separate security concern.

### 11. PM2 log rotation config uses undocumented fields

**File:** `ecosystem.config.js` lines 20-21

`max_size` and `retain` are `pm2-logrotate` module options, not native ecosystem config fields. They will be silently ignored unless `pm2-logrotate` is installed separately.

**Fix:** Document that `pm2 install pm2-logrotate` is required, or remove the fields and configure log rotation externally.

## Low Priority

### 12. Nginx: security headers on static assets location get overridden

**File:** `packages/frontend/nginx.conf` lines 18-21

The `location ~*` block for static assets uses `add_header Cache-Control` which, per Nginx behavior, replaces all parent-level `add_header` directives. This means the security headers (X-Frame-Options, X-Content-Type-Options, Referrer-Policy) defined at server level are not applied to static asset responses.

**Fix:** Repeat security headers inside the static assets location block, or use `include` with a shared snippet.

### 13. Redis connection URL hardcoded as fallback in multiple places

**Files:** `src/lib/redis.ts` line 6, `tests/setup.ts` line 10

`redis://localhost:6379` appears as fallback in both. Extract to a constant or rely solely on env config.

### 14. `as never` type casts in rate-limiter

**File:** `packages/backend/src/middleware/rate-limiter.ts` lines 10, 48

Two `as never` casts suppress type errors. These are pragmatic workarounds for library type mismatches but should be documented with comments explaining why.

## Edge Cases Found by Scouting

1. **ESL daemon auto-connect in test**: Even though `ESL_ENABLED=false`, the import chain pulls in esl-daemon module which may have side effects
2. **Socket.IO initialization during tests**: `initSocketIO(server)` runs on import -- may interfere with test isolation
3. **`getRequiredEnv` throws before test setup**: If any module is imported before vitest setupFiles run, the env vars are not set and the app crashes. Current setup order is correct but fragile.
